# Weekly Fixture

Static web page showing this week's FHC fixtures, styled after the club's social-media graphics.

Views, switched via the tabs at the top:

| Tab | Contents |
| --- | --- |
| **All** | Every fixture for the week |
| **Premier League** | Men's PL / PLR and Women's PL / PLR |
| **Seniors** | All other senior men's / women's grades (Pennant A–D, Metro 1/2, etc.) |
| **Midweek** | All midweek competitions |
| **Juniors** | All junior competitions |

The browser reads a single source — `data/season.json` — for every view: this week's homepage, week navigation backwards/forwards, and the per-team season pages. The file is regenerated daily by the `sync-calendars` GitHub Action and served directly from jsDelivr's GitHub mirror, so data refreshes do **not** trigger a Netlify redeploy (see [Hosting and data flow](#hosting-and-data-flow) below).

A smaller `data/fixtures.json` is still produced by the workflow as a backwards-compat artefact for any external consumers, but the SPA itself only loads `season.json`.

```text

```

## Run locally

From the **repo root**:

```bash
npm run dev
```

Then open <http://localhost:3000>.

## Pages and routes

| File | Purpose |
| --- | --- |
| `index.html` | Single-page app rendering the views below from `season.json`. |
| `subscribe.html` | Friendly alternative to `docs/competitions-mobile.md` — per-category calendar subscribe instructions. Linked from the fixtures page. |

The fixtures SPA uses hash routing; URLs are shareable:

| Route | Renders |
| --- | --- |
| `/` (or `#`) | Current Mon–Sun, grouped by day. Tabs (All / PL / Seniors / Midweek / Juniors) filter the list. |
| `#/week/2026-04-20` | The week starting that Monday. Tabs still filter; the `‹` / `›` arrows step ±1 week and the date label jumps back to "this week" when you've drifted. |
| `#/team/men-pl` | The selected team's full season as a continuous list. Each row carries `WED 22 APR 19:00`. The chevron `›` on each homepage row deep-links here, carrying optional `?w=…&t=…` query params so the "‹ Fixtures" back link returns to the same week + tab. |

## PWA support

The site is a [Progressive Web App](https://web.dev/progressive-web-apps/): installable on Android Chrome and "Add to Home Screen" on iOS. When installed it opens in standalone mode (no browser chrome) and works offline against the last-seen `season.json`.

- `icons/site.webmanifest` declares both `purpose: "any"` and `purpose: "maskable"` icons.
- `sw.js` precaches the app shell on install, serves cache-first for shell assets (with background refresh), and network-first for `/data/*.json` so users get the freshest fixtures when online and the last-known set offline.
- jsDelivr-hosted JSON in production is *not* cached by the SW (cross-origin opaque responses are more trouble than they're worth) — but jsDelivr's own CDN handles that.
- Bump `CACHE_VERSION` in `sw.js` whenever you ship breaking changes to shell assets so old caches are evicted.

The service worker is **not** registered on `localhost` — it would interfere with `npm run dev` file edits.

## Subscribe page data

`subscribe.html` reads `subscribe-data.json` using the same jsDelivr-in-production / `/data/...`-on-localhost split as the fixtures page. The committed file is regenerated automatically whenever you run:

```bash
npm run generate-docs
```

The generator (see [src/setup/generate-docs.js](../src/setup/generate-docs.js) — `generateSubscribeData`) pulls the same category + calendar metadata used for the mobile markdown, so the subscribe page and `docs/competitions-mobile.md` stay in sync.

## How the data is built

- `npm run scrape-scores` fetches each active competition's HV team page (`competition.fixtureUrl`) once daily and writes `temp/scores.json`. Each card on the page yields one entry: round, Melbourne-local datetime, venue abbreviation, opponent club name, status (`Played` / `Playing`), score, result badge, HV game ID. WAF blocks are tolerated — the workflow step is `continue-on-error: true`.
- `npm run generate-season-json` joins the scrape against `config/competitions.json` (team metadata, URLs) and `config/mappings-club-names.json` (opponent → abbreviation), then writes `data/season.json`. It infers home/away from the venue (FHC venues start with the configured club name), converts Melbourne local → UTC ISO via `Intl.DateTimeFormat`, and assigns each team a stable URL slug like `men-pl` / `women-pen-a` / `midweek-mens-40plus-nw`.
- `npm run generate-fixtures-json` continues to produce the older, narrower-window `fixtures.json` from the four category Google Calendars. Score merging there uses `(Melbourne local datetime, venue abbreviation)` to match scraped scores onto events. Kept for back-compat — the SPA itself reads only `season.json`.

### Season data shape

```jsonc
{
  "generatedAt": "2026-04-25T22:00:00.000Z",
  "sourcesGeneratedAt": "2026-04-25T17:00:00.000Z",
  "teams": [
    {
      "slug": "men-pl",
      "label": "MPL",
      "name": "Mens Premier League - 2026",
      "category": "mens",
      "view": "pl",
      "fixtureUrl": "https://www.hockeyvictoria.org.au/games/team/25879/409884",
      "ladderUrl": "https://www.hockeyvictoria.org.au/pointscore/25879/42156",
      "competitionUrl": "https://www.hockeyvictoria.org.au/games/25879/42156",
      "events": [
        {
          "round": 1,
          "dtstart": "2026-04-11T03:30:00.000Z",
          "dtend":   "2026-04-11T05:00:00.000Z",
          "home": "FHC",
          "away": "ALT",
          "isHome": true,
          "location": "Footscray Hockey Club - ASF",
          "gameId": "2476105",
          "score": "4-4"
        }
      ]
    }
  ]
}
```

`view` controls which homepage tab the team's fixtures appear under. `slug` is what hash-routes resolve against (`#/team/{slug}`). `score` is omitted on unplayed events.

### Front-end (`js/app.js`)

- Lazy-loads `season.json` once; subsequent route changes are pure client-side renders.
- Production fetches via jsDelivr; `localhost` reads `/data/season.json` from disk so `npm run dev` works against the working tree.
- Tabs (`All` / `PL` / `Seniors` / `Midweek` / `Juniors`) filter the active week. They hide automatically in the team-season view.
- A score (`"3-1"`, home-then-away) renders in the VS chevron slot for any played event.
- The `generatedAt` timestamp from `season.json` is rendered as a small version-tag-style stamp at the bottom of the page (e.g. `v2026.04.25-1659`, UTC). Use it to spot stale data at a glance.
- Past games are faded via `.is-past` once their end-time has passed.

## Hosting and data flow

The site is served from Netlify on the free tier, and the daily data refresh is decoupled from the deploy pipeline so we don't burn deploy credits on data-only commits.

```text
GitHub Action (daily)              Netlify                          Browser
──────────────────────             ───────                          ───────
sync-calendars.yml runs            netlify.toml `ignore` script:    On localhost:
  → npm run process-...            skip build when only files in      fetch /data/season.json
  → npm run scrape-scores          `weekly-fixture/data/` changed.  In production:
  → npm run generate-fixtures-json Code commits still deploy          fetch cdn.jsdelivr.net/gh/...
  → npm run generate-season-json   normally.                          /weekly-fixture/data/season.json
  → npm run generate-docs                                              (jsDelivr mirrors the file
  → git commit + push to main                                          from GitHub; purge step keeps
  → curl purge.jsdelivr.net/...                                        it fresh within seconds.)
```

Key files:

- [`netlify.toml`](../netlify.toml) — `[build] ignore` skips deploys when only `weekly-fixture/data/*` (or unrelated repo files) changed.
- [`.github/workflows/sync-calendars.yml`](../.github/workflows/sync-calendars.yml) — calls jsDelivr's purge endpoint after each push so the new data overrides jsDelivr's default 12-hour cache on `@main` refs.
- `js/app.js` and `js/subscribe.js` — pick the data URL based on `location.hostname`.

If you ever want to force a Netlify deploy after a data-only commit (e.g. to verify a build), make a no-op edit to any file outside `weekly-fixture/data/` (`netlify.toml`, an HTML/CSS/JS file, etc.) and push.

## Updating

- **Calendar IDs**: category calendar IDs are read from `config/settings.json` (`categoryCalendars.*.calendarId`). If a calendar is recreated, update settings.json and re-run `npm run generate-fixtures-json`.
- **Logos**: drop a new `<ABBR>.png` into `images/logos/`. Abbreviations come from `config/mappings-club-names.json`. Non-PNG extensions can be registered in `LOGO_EXTENSIONS` in `js/app.js`.
- **Styling**: edit `css/styles.css`.
- **OG image**: edit `src/og-card.html`, then run `npm run generate-og-image` from the repo root.

## Notes

- Fixtures are filtered client-side against Melbourne time, so visitors in other timezones still see the correct week.
- There is no runtime call to Google Calendar from the browser — data comes from a committed JSON file refreshed daily by the GitHub Action.
